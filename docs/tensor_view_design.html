TinyInfer TensorView Design

Design doc for the first TinyInfer tensor metadata object. This document defines the scope, invariants, memory semantics, and implementation plan for TensorView.

StatusDraft for Step 1 implementation review.
Parent designTinyInfer Core Runtime Design
OwnerCore runtime metadata layer.
Main decisionTensorView is a view, not storage.

1. Context And Scope

TinyInfer uses static arena allocation for weights, KV cache, and workspace. A tensor object that owns memory would hide allocation and lifetime in the wrong place. TensorView exists to describe a logical tensor inside a MemoryArena without owning or releasing memory. 

MemoryArena
  backend native handle: private, backend-only
  capacity: bytes
  used: bump offset

TensorView
  arena: MemoryArena*
  byte_offset: offset inside arena
  dtype: element type
  shape: logical dimensions
  strides: element strides

This design covers only the metadata layer: shape, stride, dtype, byte size, byte offset, and validity checks. It does not cover Backend memory allocation, native object access, or operator math.

2. Goals

  • Keep TensorView small, trivially movable, and cheap to pass by value or const reference.
  • Make memory ownership explicit: only MemoryArena owns memory.
  • Represent contiguous tensors and simple non-contiguous views through shape and strides.
  • Allow byte-size and storage-span checks without touching backend memory.
  • Prevent metadata bugs such as out-of-rank access, invalid shape, and byte-size overflow.
  • Keep v1 behavior simple enough to test without Metal or a real model.

3. Non-Goals

  • No memory ownership in TensorView.
  • No implicit allocation, free, copy, transpose, or materialization.
  • No public view algebra API in v1, including transpose, slice, reshape, permute, or contiguous.
  • No negative-stride view support in v1.
  • No requirement that every Backend operator accepts non-contiguous tensors.
  • No broadcasting contract in v1, even though stride metadata can technically describe overlapping views.

4. Requirements And Constraints

Area Requirement Reason
Ownership Destroying TensorView must not release memory. Arena lifetime must remain visible and easy to reason about.
Addressing TensorView stores byte_offset only. Native object access is backend-only. CPU pointers and Metal buffers have different semantics, so the core view must not interpret backend handles.
Shape Rank must be at most 8. Normal dimensions must be positive. Fixed-size arrays keep the object simple and allocation-free.
Scalar Rank-0 shape is valid and has one element. This matches normal tensor metadata semantics and keeps helper functions total.
Strides Contiguous strides are row-major. Negative strides are invalid in v1. Negative strides complicate address range checks and are not needed for LLaMA v1.
Overflow Element count, byte count, and storage span must fail closed on overflow. Overflow in memory metadata is a correctness and security bug.
Backend ops Operators may require contiguous() unless their contract says otherwise. v1 kernels should be simple, explicit, and benchmarkable.

5. Proposed Design

5.1 Data Model

struct Shape {
    uint32_t ndim;
    int64_t dims[8];
};

struct Strides {
    uint32_t ndim;
    int64_t values[8];
};

struct TensorView {
    MemoryArena* arena;
    size_t byte_offset;
    DType dtype;
    Shape shape;
    Strides strides;
};

Shape dimensions are logical element counts. Stride values are measured in elements, not bytes. byte_offset is measured in bytes from the arena base.

5.2 Shape Semantics

  • ndim == 0 is a scalar shape and has numel() == 1.
  • ndim > 8 is invalid.
  • For ndim > 0, every dimension must be positive.
  • Shape::dim(i) returns zero when i is outside the representable range.
  • Shape::numel() returns zero for invalid shapes or overflow.

5.3 Stride Semantics

  • contiguous_strides(shape) returns row-major strides.
  • Strides::stride(i) returns zero when i is outside the representable range.
  • TensorView::contiguous() compares stored strides to row-major contiguous strides.
  • storage_span_nbytes() computes the byte range touched by the view.
  • Negative strides return an invalid storage span in v1.

5.4 Transpose And View Operations

Transpose is not a memory operation at the TensorView layer. If a transpose view is added later, it should create a new TensorView with swapped shape and stride metadata: 

// Logical transpose of a 2D contiguous [M, N] tensor.
// Original shape:  [M, N]
// Original stride: [N, 1]
// Transposed shape:  [N, M]
// Transposed stride: [1, N]
// Memory copied: no

v1 does not expose a public transpose() helper because the dedicated LLaMA inference path should not need runtime transposes. Weights should be loaded or stored in the layout expected by kernels. If an operator needs a different physical layout, that conversion must be an explicit Backend operation or loader decision, not hidden inside TensorView.

Backend operators are allowed to reject non-contiguous views until an op-specific contract says otherwise. A future contiguous or layout-conversion op would allocate from workspace and copy/materialize data explicitly.

5.5 Defined View

TensorView::defined() should mean the view can safely describe a byte range inside its arena. It requires:

  • the arena pointer is non-null;
  • the arena has a backend-bound native handle;
  • the tensor metadata has a non-zero valid storage span;
  • byte_offset + storage_span_nbytes() fits inside arena capacity.

5.6 Backend Native Access

TensorView and MemoryArena do not expose public native handles. Backend implementations are the only layer that may interpret the arena's backend-native object. The view contributes only byte_offset. 

CPU backend:
  data = static_cast<uint8_t*>(arena_handle(*view.arena)) + view.byte_offset

Metal backend:
  buffer = static_cast<MTLBuffer*>(arena_handle(*view.arena))
  encoder.setBuffer(buffer, offset: view.byte_offset)

This keeps CPU pointer arithmetic out of the common tensor metadata layer and avoids treating opaque Metal objects as byte-addressable pointers.

8. Test Plan

  • Compile smoke test for tinyinfer/core.h.
  • Unit tests for dtype byte sizes.
  • Unit tests for scalar, valid, invalid, and overflow shapes.
  • Unit tests for row-major contiguous stride generation.
  • Unit tests for logical_nbytes and storage_span_nbytes.
  • Unit tests for non-contiguous gapped stride span.
  • Unit tests for negative stride rejection.
  • Compile-time tests that TensorView and MemoryArena do not expose public native_handle().
  • Later backend op tests must explicitly validate layout expectations.

9. Alternatives Considered

Alternative Decision Reason
Owning Tensor Rejected. It hides allocation and release inside the tensor object, which conflicts with arena-based memory ownership.
Contiguous-only TensorView Rejected. Shape and strides are still useful for precise layout description and future metadata-only views.
Full view algebra now Rejected for v1. Transpose, slice, permute, and reshape helpers are not required for the first dedicated LLaMA path.
Implicit materialization for transpose Rejected. Hidden copies would make hot-path memory behavior hard to reason about and hard to profile.
Negative strides Rejected for v1. They complicate address-span validation and are unnecessary for model inference layout.

10. Risks

  • TensorView can outlive its arena. Mitigation: keep long-lived views inside LlamaInferEngine and avoid exposing internal views publicly.
  • Non-contiguous metadata may be passed to an op that assumes contiguous memory. Mitigation: each op must validate layout until non-contiguous support is intentionally added.
  • Stride metadata can describe overlapping views. Mitigation: v1 does not define broadcasting or aliasing semantics for backend ops.
  • Backend code can still misuse native handles. Mitigation: keep native access protected on Backend, and keep CPU pointer arithmetic inside CPU-only code.

11. Open Questions

  • Should rank-0 scalar allocation be allowed from MemoryArena::alloc, or only represented as metadata? The current design allows scalar metadata; arena allocation can be stricter if unused.
  • Should future reshape be allowed only for contiguous views? Recommended answer: yes, unless there is a concrete non-contiguous use case.
  • Should future transpose exist as a free helper or remain an internal utility? Recommended answer: keep it internal until a real model path needs it.
