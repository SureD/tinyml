TinyInfer Core Runtime Design

Design doc for the first implementation of TinyInfer's core runtime: tensor memory, backend boundary, operator boundary, model layout, and inference engine flow.

StatusDraft, implementation-ready sections are listed below.
TargetDedicated LLaMA/TinyLlama inference runtime.
LanguageC++20 public API with explicit ownership.
Primary constraintNo hidden allocation in the decode hot path.

1. Context And Scope

TinyInfer is a dedicated LLaMA-style inference program. It is not a general tensor framework and not a graph execution engine. The first implementation should make the inference path readable, testable, and suitable for later CPU and Metal optimization.

The main design problem is memory ownership. A normal owning tensor abstraction makes ownership and allocation too implicit for this project. The selected design separates tensor metadata from storage ownership: 

DType / Shape / Strides / Device / Status
        ↓
TensorView: non-owning tensor metadata
        ↓
Backend: minimal memory allocation contract
        ↓
MemoryArena: one backend allocation, many TensorView slices
        ↓
Backend ops: explicit coarse operator boundary
        ↓
Model layout: LLaMA-specific weight and KV-cache views
        ↓
LlamaInferEngine: prefill / decode_one / generate

2. Goals

  • Use TensorView as the common data object for model weights, KV cache, workspace, and operator inputs/outputs.
  • Use MemoryArena as the only runtime memory owner in v1.
  • Allocate weights, KV cache, and workspace during LlamaInferEngine::create.
  • Keep public API small: Backend, LlamaConfig, GenerateConfig, LlamaInferEngine.
  • Make backend extension possible at coarse operator boundaries without adding a graph abstraction.
  • Make every stage independently testable before optimized Metal kernels exist.

3. Non-Goals

  • No generic computation graph.
  • No lazy evaluation.
  • No public Stream API in v1.
  • No per-tensor owning allocation/free API.
  • No tokenizer or checkpoint loader in this design.
  • No batching, continuous batching, paged attention, or speculative decoding in v1.
  • No quantization in this design.
  • No hidden hot-path heap allocation.

4. Requirements And Constraints

Area Requirement Reason
Memory Arena allocation must be explicit, move-only, and released by RAII. Prevents unclear ownership and double-free risk.
Tensor TensorView must never own memory. Allows model layout to be simple slices into preallocated arenas.
Backend Backend owns allocation, copy, op execution, and synchronization. Keeps CPU/Metal extension behind one boundary.
Engine Engine owns weights, KV cache, workspace, and inference state. The public API should be the inference program, not a set of loose utilities.
Hot path Decode must reuse workspace and append to KV cache without heap allocation. Decode is memory-bandwidth and latency sensitive.
Testing Core metadata, arena behavior, backend call order, and CPU reference math must be testable without Metal. Correctness comes before optimized kernels.

5. Proposed Design

5.1 Core Metadata

DType, Shape, Strides, Device, Status, and Result<T> are plain core types. They must not depend on model code.

  • Shape supports up to 8 dimensions.
  • All dimensions must be positive for normal tensor allocation.
  • numel returns the product of dimensions.
  • contiguous_strides returns row-major contiguous strides.
  • dtype_size maps dtype to byte size.

5.2 TensorView

TensorView is a descriptor for a logical tensor stored inside a MemoryArena. It contains arena pointer, byte offset, dtype, shape, and strides.

Detailed design: TinyInfer TensorView Design.

  • defined() is true only when the arena exists and has a native allocation.
  • logical_nbytes() is numel * dtype_size.
  • storage_span_nbytes() computes the byte range touched by the view according to shape and strides.
  • Backend code interprets the arena native handle; TensorView only exposes byte_offset.
  • Destroying a TensorView has no side effects.

5.3 Backend Memory Contract

Backend allocation is required before MemoryArena can be fully implemented. The minimal contract is:

Detailed design: TinyInfer Backend Design. 

class Backend {
public:
    virtual Device device() const = 0;
    virtual Status alloc_arena(MemoryArena& arena, size_t bytes, MemoryKind kind) = 0;

protected:
    void bind_arena(MemoryArena& arena, void* native_handle, size_t bytes, MemoryKind kind);
    void* arena_handle(const MemoryArena& arena) const;
    virtual Status release_arena(MemoryArena& arena) = 0;
};

Only Backend can bind, release, and interpret a native arena handle. This keeps native memory ownership out of TensorView and out of model code.

5.4 MemoryArena

MemoryArena owns one backend allocation. It exposes bump allocation to carve the arena into tensor views.

  • Move-only. Copy is disabled.
  • Destructor calls the owning backend's release_arena.
  • alloc(shape, dtype) returns a TensorView with runtime-managed aligned byte offset.
  • reset() sets used bytes to zero and keeps the native allocation.
  • capacity() and used() expose memory accounting for tests and profiling.

5.5 Backend Operators

Operators are explicit Backend methods, not an Op class hierarchy. Virtual dispatch is acceptable at this coarse boundary because real backends will spend most time in kernel execution or CPU loops. 

copy_from_host(dst, src, bytes)
copy_to_host(dst, src, bytes)
matmul_out(out, x, w)
rms_norm_out(out, x, weight, eps)
rope_inplace(q, k, start_pos, theta)
attention_out(out, q, k, v, k_cache, v_cache, start_pos, kv_len)
swiglu_out(out, gate, up)
argmax(out_token, logits)
synchronize()

CPU reference implementations should be written before Metal kernels. Metal can then be validated op by op against CPU behavior.

5.6 Model Layout

Model storage is internal to LlamaInferEngine. Public API should not expose NamedTensor, graph nodes, or memory planning objects.

  • weights_: token embedding, layer weights, final norm, lm head.
  • kv_cache_arena_: persistent keys and values.
  • workspace_: temporary activations reused by prefill and decode.

KV cache layout is fixed as [layer, kv_head, position, head_dim] for both keys and values. This layout keeps layer and head indexing explicit and makes later backend-specific optimization local to attention kernels.

5.7 LlamaInferEngine

LlamaInferEngine is the public inference API. It owns the backend reference, config, arenas, model views, KV cache views, workspace, logits view, and current sequence length. 

static Result<LlamaInferEngine> create(Backend& backend, const LlamaConfig& config, uint32_t max_seq_len);
Status reset();
Status prefill(std::span<const TokenId> prompt, TokenId& next_token);
Status decode_one(TokenId token, TokenId& next_token);
Status generate(std::span<const TokenId> prompt, std::span<TokenId> output, const GenerateConfig& config, uint32_t& output_count);

The inference flow should remain readable: embedding, per-layer attention path, per-layer MLP path, final norm, lm head, argmax. Allocation planning stays inside create; runtime methods should only reset and reuse workspace.

6. Implementation Plan

Step Scope Implementation Acceptance Criteria Depends On
1 Core metadata Implement dtype size, shape validation, numel, contiguous strides, tensor byte-size helpers. Unit tests pass for scalar-like edge cases, normal shapes, invalid shapes, contiguous and non-contiguous views. None
2 Backend memory contract Finalize Backend::alloc_arena, protected bind_arena, and protected release_arena. Fake backend can allocate and release a native host block through the public backend contract. Step 1
3 MemoryArena Implement move-only RAII, aligned bump allocation, reset, accounting, and native-handle offset. Tests cover move, destruction, reset, alignment, out-of-capacity errors, undefined arena errors. Step 2
4 CPU backend baseline Add CPU backend or test backend with host allocation, copy, argmax, RMSNorm, matmul, RoPE, SwiGLU, attention. Small deterministic numerical tests pass without Metal. Step 3
5 Model memory layout Compute arena sizes in LlamaInferEngine::create; bind weight, KV cache, and workspace views. Layout tests verify expected offsets, shapes, strides, and total arena usage. Step 3
6 Engine inference flow Implement prefill, decode_one, and generate using Backend ops and internal workspace views. Fake backend call-order tests prove expected high-level sequence and seq_len updates. Step 4, Step 5
7 Metal backend memory Implement Metal arena allocation, release, copy, and synchronize. Host/device copy tests pass for weights, KV cache, workspace-sized buffers. Step 3
8 Metal backend ops Replace CPU reference ops one at a time with Metal kernels. Each Metal op matches CPU reference within dtype tolerance on small and representative shapes. Step 4, Step 7
9 Profiling hooks Add optional stats and event collection for arena usage, op duration, bytes, shape, and token latency. Profiling is off by default and does not add hidden allocation to decode. Step 6

8. Test Plan

  • Compile smoke test for all public headers.
  • Core tests for DType, Shape, Strides, and TensorView byte calculations.
  • Arena tests for allocation, reset, alignment, move-only behavior, and release behavior.
  • Fake backend tests for engine call order and sequence length updates.
  • CPU backend numerical tests for each operator.
  • CPU-vs-Metal parity tests after Metal kernels exist.
  • End-to-end token parity against the Python reference after checkpoint loading exists.

9. Alternatives Considered

Alternative Decision Reason
Owning Tensor Rejected for v1. It hides allocation and lifetime in the core data object, which conflicts with static arena planning.
Public Stream Rejected for v1. Single device and single request do not need user-visible stream control. Backend can own its default queue.
Generic graph runtime Rejected. The project is a dedicated LLaMA inference engine. A graph adds more abstraction than the current scope needs.
Separate public memory plan object Rejected for v1. Memory planning belongs inside LlamaInferEngine::create until there is a real external use case.
Operator class hierarchy Rejected for v1. Backend methods are simpler and easier to inspect. Each op is already a stable coarse boundary.

10. Risks

  • Arena sizing can be wrong early. Mitigation: layout tests must verify every view offset and total used bytes.
  • TensorView can outlive its arena. Mitigation: keep views internal to engine wherever possible and avoid returning long-lived internal views publicly.
  • CPU reference ops may not match final Metal numerical behavior exactly. Mitigation: define dtype tolerance per op before parity testing.
  • Fixed KV layout may need adjustment for optimized attention. Mitigation: keep layout decision localized in model binding and attention kernels.
