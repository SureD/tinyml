TinyInfer Backend Design

Design doc for the first concrete TinyInfer backend. This document defines the backend boundary, CPU baseline implementation, operator contracts, dependencies, and acceptance criteria before optimized Metal kernels are implemented.

1. Context And Scope

TinyInfer has a fixed tensor metadata layer: TensorView describes a slice in MemoryArena, and only Backend may interpret the arena native handle. The next implementation step is to replace the fake test backend with a real backend that can allocate memory, copy bytes, and execute deterministic operator math. 

TensorView
  arena pointer + byte offset + dtype + shape + strides
        ↓
Backend
  owns native arena handle interpretation
  owns copies, operators, and synchronization
        ↓
CPUBackend
  host allocation + f32 reference kernels
        ↓
MetalBackend
  later optimized backend validated against CPUBackend

This design covers the concrete CPU backend and the shared backend operator contract. It does not implement model loading, tokenizer, quantization, batching, or Metal kernels.

2. Goals

  • Add a concrete CPUBackend that can run deterministic small numerical tests without Metal.
  • Keep Backend as the only layer that reads or writes backend-native memory.
  • Define simple v1 operator contracts: f32 math, contiguous inputs/outputs unless stated otherwise, and strict shape validation.
  • Use CPU behavior as the correctness oracle for later Metal operator parity tests.
  • Keep backend polymorphism at coarse operator boundaries only; do not introduce an Op class hierarchy.
  • Keep runtime allocation explicit: backend allocates arenas, MemoryArena carves views, ops write to caller-provided outputs.

3. Non-Goals

  • No optimized CPU SIMD kernels in the first backend implementation.
  • No thread pool or parallel execution in v1 CPU backend.
  • No public stream API.
  • No hidden workspace allocation inside backend ops.
  • No implicit dtype conversion.
  • No support for non-contiguous op inputs unless an op contract explicitly allows it later.
  • No Metal implementation work in this step beyond keeping the existing stub compiling.

4. Requirements And Constraints

Area Requirement Reason
Ownership CPUBackend::alloc_arena allocates one native host block and binds it through protected bind_arena. Keeps arena ownership under RAII and keeps native handles private to backend code.
Release release_arena must free only CPU-owned arena handles and tolerate an undefined arena. Move-only arenas and partially initialized engines must not double free or leak.
Addressing CPU pointer calculation is internal: base + TensorView::byte_offset. Core tensor code must not assume all backend handles are byte-addressable pointers.
Validation Each op validates defined views, owning backend, device type, dtype, shape, capacity span, and contiguity. Failing closed is better than producing silent numerical corruption.
Math dtype Initial CPU math supports DType::f32 only. It fixes correctness semantics before introducing f16/bf16 conversion and rounding behavior.
Hot path Ops must write to caller-provided output views and must not allocate heap memory. Decode latency should remain easy to inspect and profile.
Synchronization CPUBackend::synchronize is a no-op success. CPU execution is synchronous in v1, but the shared API keeps parity with future asynchronous backends.

5. Proposed Design

5.1 Public Factory

Add a CPU backend factory next to the existing Metal factory: 

Result<std::unique_ptr<Backend>> create_cpu_backend();
Result<std::unique_ptr<Backend>> create_metal_backend();

The factory returns the abstract Backend pointer because model code should not depend on concrete backend classes.

5.2 CPU Arena Allocation

CPUBackend stores a native host allocation in MemoryArena. The handle is treated as a base byte pointer inside CPU backend code only. 

class CPUBackend final : public Backend {
public:
    Device device() const override { return {DeviceType::cpu, 0}; }
    Status alloc_arena(MemoryArena& arena, size_t bytes, MemoryKind kind) override;
    Status copy_from_host(const TensorView& dst, const void* src, size_t bytes) override;
    Status copy_to_host(void* dst, const TensorView& src, size_t bytes) override;
    ...

protected:
    Status release_arena(MemoryArena& arena) override;
};
  • Use one host allocation per arena.
  • Use at least 64-byte alignment for cache-line-friendly base addresses.
  • Reject zero-byte allocation.
  • Bind MemoryKind without changing behavior; it is metadata for tests and later profiling.

5.3 Internal Pointer Access

CPU backend should centralize pointer calculation in private helpers: 

uint8_t* bytes(const TensorView& view);
const uint8_t* bytes(const TensorView& view) const;
float* f32_data(const TensorView& view);
const float* f32_data(const TensorView& view) const;

These helpers must validate the view before pointer use. They are implementation details, not public API.

5.4 Copy Contract

  • copy_from_host requires a defined contiguous destination view.
  • copy_to_host requires a defined contiguous source view.
  • bytes must be less than or equal to logical_nbytes().
  • Host pointers must be non-null when bytes > 0.
  • Copies are raw byte copies and do not perform dtype conversion.

5.5 Operator Implementation Scope

CPU ops are reference kernels, not optimized kernels. They establish exact shape, dtype, layout, and numerical behavior for later Metal parity tests. The implemented CPU f32 op set is: 

Implemented:
  argmax
  swiglu_out
  rms_norm_out
  matmul_out
  rope_inplace
  attention_out

All CPU math operators require f32 contiguous tensors owned by the same CPU backend. Shape mismatches return Status::invalid_argument. Unsupported dtype returns Status::unimplemented until explicit conversion semantics exist.

5.6 Common Op Validation

CPUBackend should add private validation helpers before implementing math loops. The intent is to make every op fail before touching memory. 

Status validate_f32_contiguous(const TensorView& view);
Status validate_same_shape(const TensorView& a, const TensorView& b);
Status validate_rank(const TensorView& view, uint32_t rank);
  • View must be defined.
  • View's arena must be owned by this backend.
  • View must be on DeviceType::cpu.
  • View dtype must be DType::f32 for math ops.
  • View must be contiguous in v1.
  • Output views may alias inputs only when the op contract explicitly allows it. rms_norm_out supports in-place output because it completes the row reduction before writing.

5.7 Operator Specs

Op Shape Contract Semantics Implementation Notes
argmax(out_token, logits) logits=[V] or [1,V] Return the first index of the largest f32 value. Single pass over contiguous logits. Ties keep the smaller index. Inputs are expected finite; v1 does not scan separately for NaN.
swiglu_out(out, gate, up) out, gate, and up have identical shape and element count. out[i] = silu(gate[i]) * up[i], where silu(x)=x/(1+exp(-x)). Elementwise loop. Use std::exp for the reference implementation. No allocation and no fusion in the correctness backend.
rms_norm_out(out, x, weight, eps) x=[T,H], weight=[H], out=[T,H] For each row: out[t,i] = x[t,i] * rsqrt(mean(x[t,*]^2) + eps) * weight[i]. Two loops per row: reduction, then write output. Reject non-positive eps. Use f32 accumulation for the baseline.
matmul_out(out, x, w) x=[M,K], w=[N,K], out=[M,N] or contiguous [M,...] with M*N elements. Row-major dense matrix multiply: out[m,n] = sum_k x[m,k] * w[n,k]. Scalar triple loop in row-major order. Weight layout matches Hugging Face linear weights. No bias, packing, SIMD, or transpose flag.
rope_inplace(q, k, start_pos, theta) q=[T,H,D], k=[T,KVH,D], and D is even. Apply half-split LLaMA RoPE to q and k for absolute positions starting at start_pos. theta is explicit in the Backend API so the op does not hard-code model config.
attention_out(out, q, k, v, k_cache, v_cache, start_pos, kv_len) q=[T,H,D], k/v=[T,KVH,D], per-layer cache [KVH,S,D], output [T,H*D] or [T,H,D]. Write current k/v into cache, then compute grouped-query causal attention over keys [0,kv_len). No heap score buffer. The CPU reference recomputes scores during stable softmax to keep hot-path ownership simple.

5.8 Resolved RoPE And Attention Decisions

RoPE and attention are now implemented after making the model-specific data flow explicit at the Backend boundary.

Op Decision Reason
rope_inplace(q, k, start_pos, theta) Pass theta explicitly and use half-split LLaMA rotation. Keeps backend math configurable without storing model config inside the backend.
attention_out(out, q, k, v, k_cache, v_cache, start_pos, kv_len) Engine passes per-layer cache views and current projected k/v. Keeps layer indexing in the engine and lets the backend infer GQA grouping from H / KVH.

5.9 Error Model

  • invalid_argument: bad shape, undefined view, wrong owning backend, wrong device, non-contiguous view, null host pointer, insufficient byte range.
  • unimplemented: dtype or layout intentionally unsupported in v1.
  • backend_error: host allocation failure or unrecoverable backend-specific failure.
  • synchronize: success for CPU backend.

6. Implementation Plan

Step Scope Implementation Acceptance Criteria Depends On
1 Factory and build wiring Add create_cpu_backend, infer/src/cpu_backend.cc, and CMake source entry. Public headers compile; factory returns a CPU backend with DeviceType::cpu. Current Backend API
2 Arena allocation Implement aligned host allocation, protected binding, release, and no-op synchronize. Arena tests verify allocation, capacity, device, release, move behavior, and failure on zero bytes. Step 1
3 View validation helpers Add private CPU backend helpers for defined view checks, contiguous checks, dtype checks, and byte pointer access. Unit tests prove invalid views fail before memory is touched. Step 2
4 Host copies Implement copy_from_host and copy_to_host as raw byte copies over contiguous views. Round-trip copy tests pass for f32 and byte-sized payloads. Step 3
5 Op validation Add private helpers for owned CPU f32 contiguous views, exact shape equality, and rank checks. Negative tests prove bad dtype, bad rank, shape mismatch, non-contiguous view, and foreign backend arena fail before memory access. Step 4
6 Simple elementwise/reduction ops Implement argmax, swiglu_out, and rms_norm_out with scalar f32 loops. Numerical tests pass for ties in argmax, SwiGLU expected values, RMSNorm row scaling, and invalid eps. Step 5
7 Matmul Implement row-major f32 matmul_out for [M,K] * [N,K] -> [M,N]. Small deterministic matrix tests pass and shape mismatch tests fail cleanly. Step 5
8 RoPE and attention Implement explicit-theta RoPE and per-layer-cache grouped-query causal attention. Numerical tests prove RoPE rotation, cache writes, causal masking, and head grouping. Model layout details
9 Engine test migration Use CPU backend in numerical tests while keeping fake backend for call-order tests. Fake backend proves flow; CPU backend proves math contracts. Step 7

8. Test Plan

  • Compile smoke test for tinyinfer/backend.h with both CPU and Metal factories declared.
  • CPU factory test for device type and successful construction.
  • Arena allocation tests for capacity, used bytes, runtime-managed tensor offset alignment, release, and move behavior.
  • Copy round-trip tests using host arrays and contiguous TensorView slices.
  • Negative tests for undefined view, wrong device, non-contiguous copy, unsupported dtype, shape mismatch, and byte count overflow.
  • Common op validation tests for unsupported dtype, rank mismatch, shape mismatch, non-contiguous views, and foreign backend arenas.
  • Numerical tests for argmax, swiglu_out, rms_norm_out, matmul_out, rope_inplace, and attention_out.
  • Attention tests cover per-layer cache writes, causal masking, and grouped-query head mapping.
  • Engine move tests cover internal TensorView arena rebinding after move construction.
  • Keep fake backend call-order tests for LlamaInferEngine; CPU backend tests should validate math, not control flow.

9. Alternatives Considered

Alternative Decision Reason
Implement Metal first Rejected. It mixes API design, kernel debugging, and numerical validation. CPU gives a simple oracle first.
Keep only fake backend for now Rejected. Fake backend proves call order but cannot validate memory copies or operator math.
Add an Op abstraction Rejected. Dedicated LLaMA inference needs explicit coarse functions, not a generic operator registry.
Support f16/bf16 immediately Deferred. f32 first keeps reference math easy to inspect. Lower precision can be added once parity tests exist.
Add CPU threading immediately Deferred. Single-thread scalar code is easier to validate; threading is a performance step after correctness.

10. Risks

  • Attention recomputes scores instead of storing a temporary score buffer. Mitigation: keep it as correctness baseline; add a measured scratch-buffer or SIMD optimization later if needed.
  • Engine still needs complete embedding/residual/model-loading work before CPU backend can prove full checkpoint parity. Mitigation: keep op-level CPU tests deterministic until loader tests exist.
  • Moving LlamaInferEngine can leave internal TensorView arena pointers stale if new view fields are added later. Mitigation: update rebind_views_after_move whenever model state gains a view.
  • CPU reference math may be slow. Mitigation: keep small deterministic tests first; optimize only after correctness is locked down.

11. Open Questions

  • Should CPU backend reject all non-f32 math tensors now, or allow i32/u32 only for future indexing utilities?
  • Should later CPU attention use an explicit workspace score buffer once profiling shows the scalar no-allocation baseline is too slow?
  • Should embedding lookup and residual add become backend ops, or stay model-side once checkpoint loading is added?
