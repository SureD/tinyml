compute_attention 代码讲解

这份说明对应 infer/src/cpu_backend.cc 里的 attention_score() 和 compute_attention()。 它解释这段 CPU attention 代码在算什么、每个循环的意义、为什么 attention score 被重复计算，以及一个更适合性能优化的写法应该长什么样。

一句话先讲明白

compute_attention() 做的是 Transformer 里的 causal self-attention： 当前 token 的 query 去和 KV cache 里的历史 key 做点积，得到 score； 对 score 做 softmax，得到每个历史 token 的权重； 最后用这些权重对 value 加权求和，得到 attention 输出。

score[p] = dot(q, k[p]) / sqrt(head_dim)
weight[p] = softmax(score[p])
out = sum_p weight[p] * v[p]
这份代码是一个很直接、容易 debug 的正确性版本。它的主要问题不是逻辑复杂， 而是为了省中间 buffer，反复计算了同一个 dot(q, k)。

相关代码长什么样

float attention_score(
    const float* q_row,
    const float* k_cache,
    int64_t kv_head,
    int64_t key_pos,
    int64_t cache_head_stride,
    int64_t cache_pos_stride,
    int64_t head_dim,
    float scale) const {
    const float* k_row =
        k_cache + kv_head * cache_head_stride
        + key_pos * cache_pos_stride;

    float dot = 0.0f;
    for (int64_t d = 0; d < head_dim; ++d) {
        dot += q_row[d] * k_row[d];
    }
    return dot * scale;
}

这段 helper 做什么？

它只做一件事：计算一个 query head 和一个 key row 的点积。

k_cache 是一个一维 float*，代码把它解释成 一个三维逻辑数组： 

[n_kv_heads, max_seq_len, head_dim]

这不是说内存里真的有三层数组；真正的内存还是一整条 float。 当前 CPU backend 已经要求这些 tensor 是 contiguous，所以底层可以从 dims 推导出简单 stride。 

offset =
    kv_head * cache_head_stride
  + key_pos * cache_pos_stride
  + d

在 contiguous [n_kv_heads, max_seq_len, head_dim] 布局里，内部 stride 是： 

cache_head_stride = max_seq_len * head_dim
cache_pos_stride  = head_dim

代码里取的是一整行 key，所以先算到这一行的起点： 

k_row = k_cache + kv_head * cache_head_stride
                + key_pos * cache_pos_stride

然后 k_row[d] 再取这一行里的第 d 个 float。 

void compute_attention(
    float* out,
    const float* q,
    const float* k_cache,
    const float* v_cache,
    int64_t seq_len,
    int64_t n_heads,
    int64_t n_kv_heads,
    int64_t max_seq_len,
    int64_t head_dim,
    uint32_t start_pos,
    uint32_t kv_len) const {
    const int64_t group_size = n_heads / n_kv_heads;
    const float scale = 1.0f / std::sqrt(static_cast<float>(head_dim));
    const int64_t q_token_stride = n_heads * head_dim;
    const int64_t q_head_stride = head_dim;
    const int64_t out_token_stride = n_heads * head_dim;
    const int64_t out_head_stride = head_dim;
    const int64_t cache_head_stride = max_seq_len * head_dim;
    const int64_t cache_pos_stride = head_dim;

    for (int64_t t = 0; t < seq_len; ++t) {
        const int64_t query_pos = static_cast<int64_t>(start_pos) + t;
        const int64_t valid_len =
            std::min(static_cast<int64_t>(kv_len), query_pos + 1);
        for (int64_t h = 0; h < n_heads; ++h) {
            const int64_t kv_head = h / group_size;
            const float* q_row = q + t * q_token_stride + h * q_head_stride;
            float* out_row = out + t * out_token_stride + h * out_head_stride;

            // pass 1: find max score
            float max_score = -std::numeric_limits<float>::infinity();
            for (int64_t p = 0; p < valid_len; ++p) {
                const float score = attention_score(...);
                if (score > max_score) max_score = score;
            }

            // pass 2: compute softmax denominator
            float denom = 0.0f;
            for (int64_t p = 0; p < valid_len; ++p) {
                const float score = attention_score(...);
                denom += std::exp(score - max_score);
            }

            // pass 3: weighted sum of values
            for (int64_t d = 0; d < head_dim; ++d) {
                float sum = 0.0f;
                for (int64_t p = 0; p < valid_len; ++p) {
                    const float score = attention_score(...);
                    const float weight = std::exp(score - max_score) / denom;
                    const float* v_row =
                        v_cache + kv_head * cache_head_stride
                                + p * cache_pos_stride;
                    sum += weight * v_row[d];
                }
                out_row[d] = sum;
            }
        }
    }
}

输入输出的内存布局

数据 代码里的访问方式 逻辑形状
q q + t * q_token_stride + h * q_head_stride [seq_len, n_heads, head_dim]
out out + t * out_token_stride + h * out_head_stride [seq_len, n_heads, head_dim]
k_cache k_cache + kv_head * cache_head_stride + p * cache_pos_stride [n_kv_heads, max_seq_len, head_dim]
v_cache v_cache + kv_head * cache_head_stride + p * cache_pos_stride [n_kv_heads, max_seq_len, head_dim]

这些数组都是扁平的 float*。上层已经验证它们 contiguous， 所以底层函数用局部 stride 变量把一维地址计算写清楚。

把 k_cache 当成 stride 来看

你的理解是对的：k_cache 的一维寻址就是先跳到某个 head 的起始块， 再在这个 head 里面找 key_pos，最后在这个 key row 里面找 d。

逻辑下标 stride，单位是 float 意思
kv_head cache_head_stride KV head 之间的跳跃长度，等于 max_seq_len * head_dim。
key_pos cache_pos_stride 同一个 KV head 内，位置之间的跳跃长度，等于 head_dim。
d 1 一个 key 向量内部是连续 float，下一个维度就是下一个 float。 
// 逻辑访问：
k_cache[kv_head][key_pos][d]

// 真实一维地址，单位是 float：
k_cache[
    kv_head * cache_head_stride
  + key_pos * cache_pos_stride
  + d
]

// 代码先拿到 row 起点：
const float* k_row =
    k_cache + kv_head * cache_head_stride
    + key_pos * cache_pos_stride;

// 再连续读 head_dim 个 float：
dot += q_row[d] * k_row[d];
当前 CPU backend 已经先检查 tensor 是否 contiguous，所以这里不需要把 TensorView.strides 一路传进 hot path。局部 stride 变量只是为了让 地址计算更容易读。

外层循环怎么读

第一层：遍历当前 batch 里的 token
for (t = 0; t < seq_len; ++t)
每个 t 是一个要计算 attention 的 query token。
第二层：遍历 query head
for (h = 0; h < n_heads; ++h)
每个 head 都有自己的 query，也会产生自己的 output row。
第三层：遍历 KV cache 里的历史位置
for (p = 0; p < kv_len; ++p)
每个 p 是一个可被关注的 key/value 位置。
最后一层：遍历 head_dim
for (d = 0; d < head_dim; ++d)
attention 输出本身是一个长度为 head_dim 的向量。

GQA / MQA 是怎么映射的

const int64_t group_size = n_heads / n_kv_heads;
const int64_t kv_head = h / group_size;

n_heads 是 query head 数量，n_kv_heads 是 key/value head 数量。很多 LLM 用 GQA，也就是多个 query head 共享一个 KV head。

普通 MHA

n_heads == n_kv_heads，每个 query head 都有自己的 key/value head。 

q head 0 -> kv head 0
q head 1 -> kv head 1
q head 2 -> kv head 2
GQA / MQA

n_kv_heads 小于 n_heads，多个 query head 共享 KV。 

n_heads = 8
n_kv_heads = 2
group_size = 4

q head 0,1,2,3 -> kv head 0
q head 4,5,6,7 -> kv head 1
这样做的直接收益是 KV cache 更小。decode 时 KV cache 经常是内存带宽瓶颈， 减少 KV head 数量可以降低带宽压力。

Causal mask 在哪里

const int64_t query_pos = static_cast<int64_t>(start_pos) + t;

if (p > query_pos) {
    continue;
}

query_pos 是当前 query token 在完整序列里的绝对位置。 p 是 KV cache 里的位置。

如果 p > query_pos，说明这个 key/value 来自未来 token。 自回归模型不能偷看未来，所以要跳过。

当前 query 位置 可以看的 KV 位置 不能看的 KV 位置
query_pos = 0 0 1, 2, 3, ...
query_pos = 1 0, 1 2, 3, ...
query_pos = 2 0, 1, 2 3, 4, ...

为什么 score 被算了三遍

attention score 是：

score[p] = dot(q_row, k_cache[kv_head, p]) * scale

这份代码没有把 score[p] 存下来，所以后面每次需要它时， 都重新调用 attention_score()。

第 1 遍 找 max_score

softmax 需要先减去最大值，避免 exp(score) 溢出。 

max_score = max(score[p])
第 2 遍 算 denom

计算 softmax 分母。 

denom = sum(exp(score[p] - max_score))
第 3 部分 加权 V

对 value 做加权求和，得到输出向量。 

out[d] =
  sum(weight[p] * v[p][d])
关键点：第三部分不只是又算一遍 score。因为它外面还有 for (d = 0; d < head_dim; ++d)， 所以它会对每个输出维度都重新计算所有 score。

对一个固定的 (t, h)，如果可见 KV 数量是 valid_len，当前代码里的 attention_score() 调用次数大概是：

valid_len + valid_len + head_dim * valid_len
= (head_dim + 2) * valid_len

如果 head_dim = 128，那就是 130 * valid_len 次 score 计算。理论上 score 只需要计算 valid_len 次，然后缓存起来。

为什么要先找 max_score

softmax 的数学定义是：

softmax(x_i) = exp(x_i) / sum_j exp(x_j)

但如果 x_i 很大，exp(x_i) 可能直接变成 inf。实际实现通常写成：

softmax(x_i) = exp(x_i - max_x) / sum_j exp(x_j - max_x)

减掉同一个 max_x 不会改变 softmax 结果，但是可以让指数运算更稳定。 所以第一遍扫描是为了数值稳定，不是多余的。

当前写法的性能问题

attention_score() 里真正贵的是这个循环： 

for (int64_t d = 0; d < head_dim; ++d) {
    dot += q_row[d] * k_row[d];
}

它每次都会连续读取一段 q_row 和一段 k_row。 对 CPU 来说，这主要消耗：

  • 浮点乘加计算量，尤其 head_dim 大时更明显。
  • 内存带宽和 cache，因为同一批 key row 被重复读取。
  • 函数调用和循环开销，虽然编译器可能 inline，但重复 dot product 仍然存在。

在 decode 阶段，seq_len 通常是 1，但是 kv_len 会随着上下文长度增长。上下文越长，这个重复计算越伤。

更清楚的伪代码

当前代码按数学意义可以整理成这样： 

for each query token t:
  query_pos = start_pos + t

  for each query head h:
    kv_head = h / group_size
    q_row = q[t, h]

    # 1. 找最大 score，用于稳定 softmax
    max_score = -inf
    for p in 0..kv_len:
      if p > query_pos:
        skip
      score = dot(q_row, k_cache[kv_head, p]) / sqrt(head_dim)
      max_score = max(max_score, score)

    # 2. 算 softmax 分母
    denom = 0
    for p in 0..kv_len:
      if p > query_pos:
        skip
      score = dot(q_row, k_cache[kv_head, p]) / sqrt(head_dim)
      denom += exp(score - max_score)

    # 3. 算输出向量的每个维度
    for d in 0..head_dim:
      sum = 0
      for p in 0..kv_len:
        if p > query_pos:
          skip
        score = dot(q_row, k_cache[kv_head, p]) / sqrt(head_dim)
        weight = exp(score - max_score) / denom
        sum += weight * v_cache[kv_head, p, d]
      out[t, h, d] = sum

更合理的实现形状

如果要优化，第一步通常是准备一个 scratch buffer，把每个可见位置的 score 或 weight 存下来。 

// scratch 长度至少是 kv_len
float* scores = scratch;

// pass 1: score 只算一次，并顺便找 max
float max_score = -inf;
for (int64_t p = 0; p < valid_len; ++p) {
    scores[p] = attention_score(q_row, k_cache, kv_head, p,
                                max_seq_len, head_dim, scale);
    max_score = std::max(max_score, scores[p]);
}

// pass 2: scores 原地变成未归一化的 weight
float denom = 0.0f;
for (int64_t p = 0; p < valid_len; ++p) {
    scores[p] = std::exp(scores[p] - max_score);
    denom += scores[p];
}

// pass 3: 用缓存下来的 weight 加权 V
for (int64_t d = 0; d < head_dim; ++d) {
    float sum = 0.0f;
    for (int64_t p = 0; p < valid_len; ++p) {
        const float weight = scores[p] / denom;
        const float* v_row =
            v_cache + (kv_head * max_seq_len + p) * head_dim;
        sum += weight * v_row[d];
    }
    out_row[d] = sum;
}
这个版本仍然有三段循环，因为 softmax 本来就需要 max 和 denom。 但它把最贵的 dot(q, k) 从 (head_dim + 2) * valid_len 次降到了 valid_len 次。

可以继续改进的地方

  • 避免 hot path 分配： scratch buffer 应该由 backend 或调用方预分配，不要在 compute_attention() 里面临时 malloc。
  • 把可见长度算出来： 可以用 valid_len = min(kv_len, query_pos + 1)， 这样内层循环就不需要每次判断 if (p > query_pos)。
  • 检查 GQA 参数： 这段代码假设 n_heads 能整除 n_kv_heads。 如果不能整除，kv_head = h / group_size 会出错。
  • 后续再做 SIMD： dot product 可以考虑让编译器 auto-vectorize，或者手写 SIMD。 但先去掉重复 score 计算，收益更直接。
  • value 加权的循环顺序： 当前是外层 d、内层 p。 这会让读取 v_row[d] 跨 row 访问。也可以考虑外层 p、内层 d，每次连续扫一整行 value， 对 cache 更友好。

最后总结

这段代码的本质很简单：对每个 query token、每个 query head，从 KV cache 中取出能看到的历史 key/value，计算 causal attention。

score 被重复计算的原因是：代码没有保存 score[p] 或 weight[p]。第一遍需要 max，第二遍需要 denom，第三部分需要 weight 来乘 value。为了保持实现简单，它选择了重复调用 attention_score()。

从系统工程角度看，这个版本清楚但慢。下一步最值得做的优化是引入预分配 scratch buffer，把 score 只算一次，然后再做 softmax 和 value 加权。
