TinyInfer Tokenized Inference Smoke Design

Design doc for the first text-level TinyLlama smoke test. The smoke uses a mature Python tokenizer to produce real token ids, then runs those ids through the C++ TinyInfer engine.

1. Context And Scope

The loader can now load TinyLlama's real config and safetensors checkpoint. A hard-coded token id sequence only proves that the numeric path does not crash. It does not prove that the model is being prompted with real text or that generated token ids can be decoded back into readable output.

The next smoke test should use the model's actual tokenizer files. To avoid spending time on tokenizer implementation before the inference path is validated, v1 should use a mature Python tokenizer library as the orchestration layer. 

prompt text
        ↓
Python tokenizers.Tokenizer.from_file(tokenizer.json)
        ↓
real token ids
        ↓
C++ TinyInfer engine
  load config + safetensors
  prefill + decode token ids
        ↓
generated token ids
        ↓
Python tokenizer.decode(...)
        ↓
readable generated text

This design covers a smoke/integration workflow only. It does not add tokenizer APIs to the C++ runtime and does not implement a full user-facing CLI.

2. Goals

  • Use TinyLlama's real tokenizer files to encode a fixed prompt into real token ids.
  • Run those token ids through the C++ engine loaded with real TinyLlama weights.
  • Decode generated token ids back into text with the same tokenizer.
  • Keep tokenizer code outside the C++ runtime for now.
  • Use a mature library instead of implementing SentencePiece/BPE logic manually.
  • Make the smoke deterministic: greedy argmax, fixed prompt, fixed max token count.
  • Keep the test manual, not part of default ctest, because scalar CPU inference is slow.

3. Non-Goals

  • No C++ tokenizer implementation in this step.
  • No interactive prompt loop.
  • No chat UI and no streaming output.
  • No sampling, temperature, top-p, top-k, repetition penalty, or beam search.
  • No attempt to make CPU inference fast.
  • No default CTest dependency on the 2GB checkpoint.
  • No full tokenizer abstraction in infer/include yet.

4. Requirements And Constraints

Area Requirement Reason
Tokenizer Use Python tokenizers with local tokenizer.json. The reference environment already depends on tokenizers, and it avoids reimplementing tokenizer internals.
Prompt Use a fixed text prompt checked into the script. The goal is deterministic smoke validation, not user interaction.
C++ boundary C++ executable should accept token ids and return generated token ids. Keeps C++ runtime focused on inference while Python owns text/token conversion.
Generation Use greedy argmax only. Matches the current LlamaInferEngine API and keeps output deterministic.
Runtime cost Keep the default generated token count small. Current CPU backend is scalar f32 and decode is expected to be slow.
Build Do not require Python tokenizer packages to build the C++ library. Tokenizer orchestration is a smoke tool, not a core runtime dependency.

5. Proposed Components

5.1 User-Facing Entry

./scripts/tinyinfer "Hello, my name is"

This is the command humans should run. It hides tokenizer setup, C++ runner arguments, and token id plumbing. If no prompt argument is provided, it asks for one interactively: 

./scripts/tinyinfer
prompt> Hello, my name is

Internally it runs through the existing Python reference environment with uv, uses the real tokenizer, calls the C++ token runner, decodes the result, and prints text.

5.2 Python Prompt Driver

scripts/tinyinfer_prompt.py

This script owns the actual prompt flow: parse text, encode tokens, call the C++ runner, decode generated tokens, and print a concise answer. It is not a unit test; it is the simple manual inference entry.

5.3 Tokenized Smoke Driver

scripts/run_tokenized_smoke.py

This remains a more verbose diagnostic path. It prints prompt ids, generated ids, elapsed times, and decoded text for debugging.

5.4 C++ Token Runner

tests/cpp/run_token_ids_smoke.cc

The C++ executable should load the model and run inference from token ids passed by the caller. It should print generated token ids in a machine-readable line. 

build/run_token_ids_smoke \
  models/TinyLlama-1.1B-Chat-v1.0 \
  128 \
  8 \
  1 15043 29892 ...

Proposed arguments:

  • argv[1]: model directory.
  • argv[2]: max sequence length.
  • argv[3]: max new tokens.
  • argv[4...]: prompt token ids.

Proposed output: 

prompt_tokens: 1 15043 29892 ...
generated_tokens: 29871 13 ...
all_tokens: 1 15043 29892 ... 29871 13 ...
elapsed_sec: 42.0

5.5 Script Layers

./scripts/tinyinfer
  ↓
uv run --project reference python scripts/tinyinfer_prompt.py
  ↓
build/run_token_ids_smoke

6. Tokenizer Choice

v1 should use tokenizers.Tokenizer.from_file(model_dir / "tokenizer.json"). This avoids requiring transformers and uses the dependency already present in the reference project.

If tokenizer.json turns out not to reproduce Hugging Face behavior exactly enough, the fallback is Python sentencepiece with tokenizer.model. That should remain a smoke-tool dependency, not a core C++ runtime dependency.

The smoke should explicitly add BOS when needed. TinyLlama config uses bos_token_id = 1 and eos_token_id = 2. The script should print whether the encoded prompt already includes BOS.

7. Fixed Prompt

Use a short English prompt first, because tokenization and model behavior are easier to inspect: 

prompt = "Hello, my name is"

This prompt should produce real tokenizer ids and a readable continuation. Later prompts can test chat formatting, but chat templates are intentionally deferred.

8. Algorithm

8.1 Python Driver

load tokenizer.json
encode fixed prompt
ensure / prepend BOS if required
run build/run_token_ids_smoke with token ids
parse generated_tokens line
decode prompt tokens, generated tokens, and all tokens
print text output

8.2 C++ Runner

parse model_dir, max_seq_len, max_new_tokens, prompt token ids
create CPU backend
load engine from model_dir
engine.prefill(prompt, next)
append next
repeat max_new_tokens - 1:
  engine.decode_one(previous, next)
  append next
  stop if next == eos_token_id
print generated token ids

9. Validation

  • C++ runner returns non-empty generated token ids.
  • Every generated token id is less than vocab_size.
  • Python tokenizer decodes generated ids without error.
  • Output text is human-readable enough to inspect manually.
  • Default unit tests still run without Python tokenizer dependencies.

10. Risks And Mitigations

Risk Mitigation
tokenizer.json behavior may differ from full Transformers chat formatting. Start with a plain completion prompt. Defer chat template support.
Scalar CPU inference may be slow for multiple generated tokens. Default to a small max-new-token count and keep this manual.
Decoded text may be odd even if math is correct. Use deterministic token output now; add Python reference parity next.
Python smoke dependencies may not be installed. Run through the existing reference uv environment.

11. Acceptance Criteria

  • Design is implemented as a manual smoke path, not default ctest.
  • ./scripts/tinyinfer "..." is the preferred manual entry.
  • Python encodes a real prompt using the local TinyLlama tokenizer.
  • C++ engine generates at least one token from the real prompt tokens.
  • Python decodes generated token ids back to text.
  • The smoke output includes prompt text, prompt token ids, generated token ids, decoded continuation text, and elapsed time.

12. Deferred Work

  • Native C++ tokenizer API.
  • Chat template support from tokenizer_config.json.
  • Python reference parity for logits or generated token sequence.
  • Sampling controls and EOS/stopping configuration from generation_config.json.
  • Interactive CLI after correctness is established.
