TinyInfer Safetensors Config Loader Design

Design doc for loading Hugging Face LLaMA config and single-file safetensors checkpoints into the existing TinyInfer engine. The first target is TinyLlama-1.1B-Chat-v1.0 with f32 runtime weights.

StatusDraft for implementation review.
Parent designTinyInfer Core Runtime Design
Depends onTensorView Design, Backend Design
Main decisionLoad BF16/F32 checkpoint tensors into existing f32 TensorViews.

1. Context And Scope

TinyInfer can now create a LLaMA-style engine, allocate arenas, bind model weight views, run token-driven prefill/decode, and execute CPU f32 correctness kernels. The remaining blocker for real inference is that the weight arena is empty. We need a loader that reads a Hugging Face model directory and fills the already-bound engine weights. 

Hugging Face model directory
  config.json
  model.safetensors
        ↓
load LlamaConfig
        ↓
LlamaInferEngine::create(...)
  alloc weights / KV cache / workspace
  bind internal TensorView weight layout
        ↓
load safetensors tensors
  validate key + dtype + shape + byte range
  convert BF16/F32 to f32
  backend.copy_from_host(...)
        ↓
engine ready for token-level inference

This design covers local file loading only. It does not cover downloading checkpoints, tokenization, sharded checkpoints, quantization, or optimized backend-specific weight packing.

2. Goals

  • Parse Hugging Face config.json into LlamaConfig.
  • Validate that the config is compatible with the current LLaMA runtime path.
  • Parse a single model.safetensors file without loading the whole file into memory.
  • Map expected Hugging Face tensor names to internal engine weight TensorViews.
  • Load BF16 and F32 checkpoint data into f32 runtime tensors.
  • Keep model memory ownership inside LlamaInferEngine; the loader only fills existing views.
  • Fail closed on missing keys, wrong shapes, unsupported dtypes, malformed files, or out-of-range offsets.
  • Add small deterministic C++ tests that do not require the 2GB TinyLlama checkpoint.

3. Non-Goals

  • No tokenizer implementation.
  • No Hugging Face download logic.
  • No support for sharded safetensors or model.safetensors.index.json in v1.
  • No support for quantized checkpoints.
  • No runtime transpose, implicit layout conversion, or weight packing.
  • No PyTorch, Python, or external safetensors runtime dependency.
  • No custom JSON parser unless the selected third-party library becomes a real build problem.
  • No support for arbitrary transformer architectures beyond the current LLaMA-style decoder path.
  • No change to CPU/Metal operator contracts in this step.

4. Requirements And Constraints

Area Requirement Reason
Ownership The loader must not allocate model tensors. It fills LlamaInferEngine's existing weight views. The engine already owns static memory planning and view binding.
Layout Loader must not transpose Hugging Face linear weights. Current matmul_out consumes weights as [N,K], matching HF [out_features,in_features].
DType Runtime weights are f32 in v1, regardless of checkpoint dtype. Current CPU kernels and weight arena are f32-only correctness paths.
Memory Read one tensor at a time, not the full safetensors data section. The TinyLlama checkpoint is about 2GB BF16; full-file buffering is unnecessary.
Validation Validate tensor key, dtype, rank, shape, byte count, and file range before copying. Checkpoint loading bugs otherwise become silent numerical corruption.
Backend boundary Use backend.copy_from_host for all writes to TensorViews. The loader should not interpret backend-native arena handles.
Dependencies Use a small pinned header-only JSON library through CMake FetchContent, and keep the rest of the loader in C++20 standard library code. JSON parsing is not the core system problem. A fixed release tag avoids reinventing a parser while keeping dependency choice explicit.

5. Proposed API

Add a small public loader API. The full-directory helper is the normal path. The split config/weights functions remain available for tests and debugging.

5.1 Example

auto backend = create_cpu_backend();

    *backend.value,

    2048);

5.2 Engine Access

LlamaInferEngine::Model should remain private. The loader needs access to internal weight views, but callers should not. The proposed implementation adds a narrow friend declaration: 

friend Status load_llama_safetensors(
    LlamaInferEngine& engine,
    const char* path);

This keeps the public inference API small and avoids exposing individual weight accessors that would become long-term compatibility baggage.

6. Config Loading

The loader reads Hugging Face LLaMA config fields and maps them into LlamaConfig. Unknown fields are ignored unless they imply unsupported behavior.

6.1 Compatibility Checks

Field Accepted value Reason
model_type"llama"The engine implements a LLaMA-style decoder path.
hidden_act"silu"The MLP path uses SwiGLU with SiLU.
attention_biasfalseCurrent projection kernels have no bias input.
tie_word_embeddingsfalseThe engine binds embedding and lm_head as separate TensorViews.
pretraining_tp1No tensor-parallel checkpoint reconstruction in v1.
rope_scalingnull or absentCurrent RoPE kernel supports base theta only.

After field mapping, the loader calls LlamaConfig::validate(). This catches internal invariants such as hidden_size % n_heads == 0, n_heads % n_kv_heads == 0, and even head_dim for RoPE.

7. Safetensors Header Loading

Safetensors starts with an 8-byte little-endian header length, followed by a JSON header, followed by raw tensor bytes. 

byte[0:8]        uint64 little-endian header_len
byte[8:8+N]      JSON header
byte[8+N:end]    raw tensor data

Each tensor entry contains dtype, shape, and data_offsets. Offsets are relative to the start of the raw tensor data section, not the start of the file. 

"model.embed_tokens.weight": {
  "dtype": "BF16",
  "shape": [32000, 2048],
  "data_offsets": [131072000, 262144000]
}

7.1 Header Validation

  • header_len must fit in memory and must not exceed file size.
  • Header JSON must parse as an object.
  • __metadata__ is allowed but is not a tensor entry.
  • Every tensor entry must contain dtype, shape, and data_offsets.
  • data_offsets must contain two non-negative integers with begin <= end.
  • end - begin must equal numel(shape) * safetensors_dtype_size(dtype).
  • 8 + header_len + end must not exceed file size.
  • All offset and byte-size arithmetic must check overflow.

8. Weight Mapping

The mapping is fixed by the engine's internal model layout. For TinyLlama, head_dim = hidden / n_heads and kv_dim = n_kv_heads * head_dim.

Safetensors key Engine TensorView Expected shape
model.embed_tokens.weightmodel_.token_embedding[vocab, hidden]
model.norm.weightmodel_.final_norm[hidden]
lm_head.weightmodel_.lm_head[vocab, hidden]
model.layers.{i}.input_layernorm.weightlayers[i].attn_norm[hidden]
model.layers.{i}.self_attn.q_proj.weightlayers[i].q_proj[hidden, hidden]
model.layers.{i}.self_attn.k_proj.weightlayers[i].k_proj[kv_dim, hidden]
model.layers.{i}.self_attn.v_proj.weightlayers[i].v_proj[kv_dim, hidden]
model.layers.{i}.self_attn.o_proj.weightlayers[i].o_proj[hidden, hidden]
model.layers.{i}.post_attention_layernorm.weightlayers[i].ffn_norm[hidden]
model.layers.{i}.mlp.gate_proj.weightlayers[i].gate_proj[intermediate, hidden]
model.layers.{i}.mlp.up_proj.weightlayers[i].up_proj[intermediate, hidden]
model.layers.{i}.mlp.down_proj.weightlayers[i].down_proj[hidden, intermediate]

Loading is strict. All expected keys must exist. Tensor shape must exactly match the destination view. Unexpected keys are errors except explicitly ignored RoPE cache metadata such as *.self_attn.rotary_emb.inv_freq.

9. DType Conversion And Memory Behavior

The v1 runtime uses f32 weights. TinyLlama's checkpoint uses BF16. The loader converts supported input dtypes to f32 before copying.

Safetensors dtype v1 behavior Notes
F32Read into f32 staging buffer and copy.No numerical conversion.
BF16Read uint16 payload, expand to f32 staging buffer, then copy.Required for TinyLlama.
F16Return unimplemented.Can be added after BF16 path is verified.
 
uint16_t bf16 = ...;
uint32_t bits = static_cast<uint32_t>(bf16) << 16;
float out;
std::memcpy(&out, &bits, sizeof(float));

The first implementation uses one tensor-sized staging buffer. The largest TinyLlama tensor is 32000 * 2048 elements, which is 256MB as f32. This is acceptable for a correctness-first loader and keeps the implementation simple.

If peak host memory becomes a problem, the same design can be extended to row-chunk loading by constructing contiguous subviews and copying smaller ranges.

10. Loading Algorithm

load_llama_from_hf_dir(backend, model_dir, max_seq_len):
  config = load_llama_config_json(model_dir + "/config.json")
  engine = LlamaInferEngine::create(backend, config, max_seq_len)
  load_llama_safetensors(engine, model_dir + "/model.safetensors")
  return engine

load_llama_safetensors(engine, path):
  parse safetensors header
  expected = build_expected_weight_table(engine)

  for each expected tensor:
    entry = header.find(expected.key)
    validate dtype, shape, byte range
    read raw bytes for this tensor
    convert to f32 staging buffer if needed
    backend.copy_from_host(expected.view, staging.data(), expected.view.logical_nbytes())

  validate no unexpected tensor keys remain
  backend.synchronize()
  return success

The loader never computes model outputs and never mutates KV cache or workspace. Its only side effect is writing the weight arena through the backend copy API.

11. JSON Library

JSON parsing is support code, not the interesting part of the inference system. The loader should use a third-party JSON library for correctness and developer speed instead of maintaining a custom parser.

The proposed dependency is nlohmann/json. It is header-only, widely used, easy to integrate through CMake, and sufficient for both Hugging Face config.json and safetensors JSON headers. 

FetchContent_Declare(
    nlohmann_json
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG v3.11.3
    GIT_SHALLOW TRUE
)

#include <nlohmann/json.hpp>

The implementation should still wrap direct JSON access in small helper functions such as require_u32, optional_f32, require_string, and require_bool. The library handles parsing; TinyInfer code still owns schema validation and error mapping to Status.

The dependency should be pinned to a fixed released tag. First configure requires network access unless the CMake dependency cache already has the source; normal rebuilds then use the cached copy under the build directory.

12. Error Handling

The implementation continues using the existing Status and Result<T> types.

Failure Status
File cannot be opened or readinvalid_argument
Malformed JSON or safetensors headerinvalid_argument
Unsupported config semanticsinvalid_config
Missing key, unexpected key, wrong shape, wrong byte rangeinvalid_argument
Unsupported checkpoint dtypeunimplemented
Backend copy failureReturn backend status unchanged

The current Status stores const char*, so v1 errors use stable fixed messages. If loader debugging becomes painful, upgrading Status to carry owned error text should be a separate change.

13. Alternatives Considered

Alternative Decision Reason
Expose public weight TensorView accessors on LlamaInferEngine Rejected Leaks internal model layout into public API and makes future weight packing harder.
Loader allocates its own weight tensors Rejected Conflicts with current arena ownership model and static memory planning.
Transpose weights while loading Rejected Current matmul already matches Hugging Face layout. A transpose would add confusion and memory cost.
Write a custom JSON parser Rejected JSON parsing is not the point of this runtime. A third-party parser reduces bugs and keeps loader work focused.
Use nlohmann/json for config and safetensors header JSON Accepted Header-only, convenient, widely used, and not on the inference hot path.
Use an external safetensors C++ library Rejected for v1 The binary container is simple enough to parse directly, and manual byte-range validation is central to this loader.
Store BF16 weights in runtime arena immediately Deferred Requires kernel and memory-plan changes. Correctness-first f32 loading is simpler.

14. Risks And Mitigations

Risk Mitigation
F32-expanded TinyLlama weights require about 4GB. Accept for v1 correctness. Later add BF16/F16 weight arena and kernels.
Manual safetensors parsing can mishandle offsets. Strict byte-range validation and small fixture tests for malformed headers.
Status messages may be too generic for large checkpoints. Start with stable messages; upgrade error type separately if needed.
Single tensor staging buffer can still be large. Keep implementation simple first; row-chunk loading is a compatible follow-up.
Config loader may accept unsupported future LLaMA variants. Fail closed on known unsupported semantics such as bias, rope scaling, and tied embeddings.

15. Test Plan

  • Parse a minimal valid config.json and compare every LlamaConfig field.
  • Reject missing required config fields.
  • Reject unsupported config values such as attention_bias=true or rope_scaling object.
  • Parse a tiny safetensors file with F32 tensors and load into CPU backend.
  • Parse a tiny safetensors file with BF16 tensors and verify f32 expansion values.
  • Reject missing weight keys, unexpected keys, wrong shapes, wrong byte ranges, and unsupported dtypes.
  • Load a complete demo-model safetensors fixture matching LlamaConfig::demo().
  • Run cmake --build build and ctest --test-dir build --output-on-failure.

The real 2GB TinyLlama checkpoint should be used as a manual or integration test, not as a default unit test dependency.

16. Acceptance Criteria

  • load_llama_config_json loads the local TinyLlama config and matches LlamaConfig::tinyllama_1_1b().
  • load_llama_from_hf_dir creates an engine and fills all expected weights from local models/TinyLlama-1.1B-Chat-v1.0.
  • Small C++ fixture tests cover config parsing, safetensors header parsing, dtype conversion, and strict key/shape validation.
  • No public API exposes raw model weight internals.
  • No operator implementation changes are required for loading.
  • The only third-party dependency is nlohmann/json, fetched by CMake from fixed tag v3.11.3.

17. Implementation Plan

  1. Fetch pinned nlohmann/json with CMake FetchContent and link its imported target.
  2. Add small JSON schema access helpers for fixed Status error mapping.
  3. Add model_loader.h public declarations.
  4. Implement load_llama_config_json and config validation.
  5. Add safetensors header parser and tensor metadata validation.
  6. Add expected weight table builder using engine internal TensorViews.
  7. Implement F32 and BF16 tensor loading into f32 staging buffers.
  8. Add CMake entries and C++ tests.
  9. Manually load the local TinyLlama checkpoint on CPU.

18. Deferred Work

  • Tokenizer integration.
  • End-to-end prompt text generation.
  • Sharded safetensors support.
  • F16 input conversion.
  • BF16/F16 runtime weight storage.
  • Chunked loading to lower peak staging memory.
  • Metal-specific upload and packed weight layout.
  • Python reference parity tool comparing logits for fixed token inputs.
