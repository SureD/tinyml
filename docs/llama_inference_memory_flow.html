Llama 推理流程与内存视角

这份笔记用 TinyInfer 当前的 llama.cc 为线索，从一个 token id 开始，解释它如何变成 hidden 向量，如何经过每一层 Transformer block 的 attention 和 FFN，最后如何得到下一个 token。

主线token id → embedding → layers → logits → next token
内存视角weights 长期只读，KV cache 跨 token 保存，workspace 每次 forward 复用。
代码入口LlamaInferEngine::run_layers()
重点形状[T,H]、[T,NH,D]、[KVH,S,D]。

先建立符号

推理代码里大量计算都可以用几个维度描述。先把这些符号记住，后面看 shape 会顺很多。

符号 代码字段 含义
T tokens.size() 本次 forward 的 token 数。prefill 时是 prompt 长度，decode 时通常是 1。
H hidden_size 每个 token 的隐藏向量长度，也是模型主干宽度。
I intermediate_size FFN 中间层宽度，通常比 H 大。
V vocab_size 词表大小，logits 的最后一维。
NH n_heads Query attention head 数。
KVH n_kv_heads Key/Value head 数。GQA 下它小于 NH。
D head_dim() 每个 head 的维度，D = H / NH。
KVD n_kv_heads * head_dim() 所有 KV heads 拼起来的宽度，KVD = KVH * D。
S max_seq_len KV cache 最大可保存的序列长度。

三块核心内存

从内存角度理解推理，最重要的是先分清三类数据的生命周期。TinyInfer 里正好把它们分成三个 arena。

weights_ 模型参数。加载后长期存在，推理时主要只读。比如 embedding table、Q/K/V/O projection、FFN projection、norm weight、lm_head。
kv_cache_arena_ 一次 request/generation 的历史状态。每生成一个 token，每一层都会把新 K/V 写进去，后续 token 的 attention 会继续读它。
workspace_ 本次 forward 的临时 buffer。prefill() 和 decode_one() 进入 run_layers() 前都会 reset，然后顺序切出 hidden、q、k、v、FFN 中间结果等。
对齐 padding arena 是一大块内存，但每个 tensor 的起始地址要对齐。padding 通常插在 tensor 之间，memory plan 会为这些间隙预留空间。

weights: 模型参数，生命周期 = engine kv_cache: 历史 K/V，生命周期 = request/generation workspace: 临时激活，生命周期 = 单次 run_layers()

总流程

run_layers() 是推理主干。它拿到一段 token ids，输出最后一个位置的 logits。

1. token ids 输入是一段整数 token id：[T]。这些整数本身没有语义向量，只是词表里的行号。
2. embedding lookup 从 token_embedding 表里按行拷贝，得到 hidden=[T,H]。
3. 重复 N 层 Transformer block 每层做 attention 子层和 FFN 子层，每个子层都有 RMSNorm、权重投影和 residual add。
4. final norm + lm_head 最后一层输出再做 RMSNorm，取最后一个 token 的 hidden，乘 lm_head 得到 logits=[1,V]。
5. sample 或 argmax 当前实现用 argmax() 选出最大 logit 的 token id，作为下一个 token。

token 如何变成 hidden 向量

token id 是一个整数，例如 42。模型不能直接对整数做 attention，所以第一步是查 embedding table：

token_embedding: [V,H] tokens: [T] hidden: [T,H] hidden[t] = token_embedding[tokens[t]]

这一步不是矩阵乘法，而是按行拷贝。CPU backend 里 embedding_out() 做的就是： 

const float* src = table_data + token * hidden;
float* dst = out_data + t * hidden;
memcpy(dst, src, hidden * sizeof(float));
内存归属： token_embedding 属于 weights；hidden 属于 workspace。一个 token 进入模型以后，后续所有主干计算基本都围绕这个 hidden 向量变换。

Transformer Block 的骨架

每一层的输入和输出都是 [T,H]。这点很重要：block 内部会把 hidden 展开成 Q/K/V 或 FFN 中间维度，但最终都会回到 [T,H]，这样下一层才能继续接上。 

// layer input
hidden: [T,H]

// attention sublayer
normed   = rms_norm(hidden, attn_norm)
q        = matmul(normed, q_proj)
k        = matmul(normed, k_proj)
v        = matmul(normed, v_proj)
q, k     = rope(q, k)
attn_out = attention(q, k, v, kv_cache)
ffn_out  = matmul(attn_out, o_proj)
ffn_out += hidden

// FFN sublayer
normed  = rms_norm(ffn_out, ffn_norm)
gate    = matmul(normed, gate_proj)
up      = matmul(normed, up_proj)
swiglu  = silu(gate) * up
hidden  = matmul(swiglu, down_proj)
hidden += ffn_out

// layer output
hidden: [T,H]

当前代码变量名里 ffn_out 有一点容易误解：attention 的 o_proj 输出先放进 ffn_out，再加 residual，随后它作为 FFN 子层的输入残差。

Attention 子层

RMSNorm

attention 前先做 RMSNorm：

hidden: [T,H] attn_norm weight: [H] normed: [T,H]

attn_norm 是权重，normed 是 workspace 中的临时激活。RMSNorm 不改变 shape，只改变数值尺度。

Q/K/V projection

然后用三组权重把 normed 投影成 Query、Key、Value：

tensor weight shape output shape 为什么是这个形状
q_proj [H,H] q=[T,NH,D]，元素数等于 [T,H] Query 有 NH 个 heads，NH * D = H，所以输出总宽度还是 H。
k_proj [KVD,H] k=[T,KVH,D] GQA 下 KV heads 更少，KVD = KVH * D，所以 K 比 Q 小。
v_proj [KVD,H] v=[T,KVH,D] Value 和 Key 使用同样的 KV head 数。

CPU backend 的 matmul_out() 约定是 x=[M,K]，w=[N,K]，输出元素数是 M*N。所以 q_proj=[H,H] 会把 normed=[T,H] 变成总元素布局为 [T,H] 的 Q，再用三维 shape 解释为 [T,NH,D]。

RoPE

RoPE 作用在 Q 和 K 上，注入位置信息：

q: [T,NH,D] k: [T,KVH,D] start_pos: 当前 token 在整段序列中的起始位置

decode 时 start_pos 不是 0，而是当前 KV cache 已经保存的长度。这样新 token 的位置编码能接在历史序列后面。

写 KV cache，再读历史做 attention

当前层 attention 会先把本次 token 的 K/V 写进该层 KV cache：

layer_k_cache: [KVH,S,D] layer_v_cache: [KVH,S,D] k current: [T,KVH,D] v current: [T,KVH,D]

写完以后，attention 会让每个 query 位置读取它允许看到的历史 K/V。为了保持 causal 语义，位置 query_pos 最多只能看到 0..query_pos，不能看未来位置。

query_pos = start_pos + t valid_len = min(kv_len, query_pos + 1) score = dot(q_head, k_cache_head) / sqrt(D) prob = softmax(score) out = sum(prob * v_cache_head)

GQA 的关键： 如果 NH=32、KVH=4，那么 8 个 query heads 共享 1 个 KV head。这样 KV cache 体积和 decode 时读取历史 K/V 的带宽都会下降。

O projection 和 residual

attention 输出是 [T,H]。随后经过 o_proj=[H,H]，仍然得到 [T,H]，最后加回 attention 子层输入：

attn_out: [T,H] o_proj: [H,H] ffn_out: [T,H] ffn_out += hidden

FFN 子层

FFN 是每个 token 独立做的 MLP，不跨 token 混合信息。跨 token 的信息混合已经在 attention 里完成了。

1. RMSNorm ffn_out=[T,H] 经过 ffn_norm=[H] 得到 normed=[T,H]。
2. 两条上投影分支 gate_proj=[I,H] 产生 gate=[T,I]，up_proj=[I,H] 产生 up=[T,I]。
3. SwiGLU 对每个元素做 silu(gate) * up，得到 swiglu=[T,I]。
4. 下投影 down_proj=[H,I] 把 [T,I] 压回 [T,H]。
5. residual add 加回 FFN 子层输入 ffn_out=[T,H]，得到下一层要用的 hidden=[T,H]。

gate = normed @ gate_proj up = normed @ up_proj swiglu = silu(gate) * up hidden = swiglu @ down_proj hidden += ffn_out

性能直觉： FFN 的大头是两次 [T,H] x [I,H] 上投影和一次 [T,I] x [H,I] 下投影。prefill 时 T 大，矩阵乘更容易吃满算力；decode 时 T=1，小矩阵乘更容易受内存带宽和调度开销影响。

最后如何得到 next token

所有层跑完以后，hidden 仍然是 [T,H]。当前实现只需要最后一个位置的下一个 token，所以取最后一行：

hidden: [T,H] last_hidden: [1,H]

然后做 final RMSNorm，再乘 lm_head=[V,H]：

logits: [1,V] logits = last_hidden @ lm_head

logits[v] 表示下一个 token 是词表里第 v 个 token 的分数。TinyInfer 当前用 argmax()，直接选择分数最高的 token。

prefill 和 decode 的区别

两者走的是同一个 run_layers()，但 T、start_pos 和 KV cache 行为不同。

阶段 输入 T KV cache 主要成本
prefill(prompt) prompt 长度，通常大于 1 从空 cache 开始，一次写入 prompt 的每个位置。 大矩阵乘和 attention。因为 T 大，更像吞吐型计算。
decode_one(token) 通常是 1 只写入新 token 的 K/V，但 attention 要读取所有历史 K/V。 KV cache 读取带宽、attention 扫历史长度、小 batch matmul 的开销。

prefill: T = prompt_len start_pos = 0 kv_len = prompt_len decode: T = 1 start_pos = cache.seq_len kv_len = cache.seq_len + 1

为什么 KV cache 重要： 没有 KV cache 时，每生成一个新 token 都要重新计算所有历史 token 的 K/V。KV cache 把历史 K/V 存下来，decode 时只计算新 token 的 K/V，然后读历史 cache 做 attention。

MemoryPlan 怎么对应这些 tensor

build_memory_plan() 做的是静态预算。它不真的分配 tensor，只是提前算出 arena 要多大。

weights

global weights: token_embedding [V,H] final_norm [H] lm_head [V,H] per-layer weights: attn_norm [H] q_proj [H,H] k_proj [KVD,H] v_proj [KVD,H] o_proj [H,H] ffn_norm [H] gate_proj [I,H] up_proj [I,H] down_proj [H,I]

KV cache

keys [L,KVH,S,D] values [L,KVH,S,D] kv_elems = 2 * L * KVH * S * D

workspace

rough scratch budget: 7 * S * H // hidden-like buffers 3 * S * I // FFN intermediate buffers S * V // logits budget, conservative in current code

这里的 workspace 预算偏保守。比如实际 logits 当前只需要 [1,V]，但 memory plan 里按 [S,V] 预留。保守预算的好处是简单，不容易不够；坏处是可能浪费内存。

工程边界： shape 和 tensor 数量是 Llama 模型知识，放在 llama.cc 合理。alignment 和 padding 是 allocator/backend 知识，后续如果支持 GPU，最好由 backend 或 arena 提供真实对齐规则。

硬件视角的直觉

  • embedding 主要是内存读取和拷贝，访问模式由 token id 决定。
  • projection matmul 是主要算力来源，prefill 时 T 大，更容易利用 SIMD/GPU。
  • decode attention 常常受 KV cache 读取带宽影响，因为每个新 token 都要扫描越来越长的历史 K/V。
  • GQA 通过减少 KVH 降低 KV cache 大小和历史读取带宽。
  • workspace arena 避免 hot path 里反复 heap allocation，内存布局也更容易 debug 和 profile。
  • 对齐 让 tensor 起始地址更适合 cache line、SIMD load、GPU vectorized load。

读代码建议

建议按这个顺序读代码，能把“数据流”和“内存流”同时串起来：

  1. LlamaInferEngine::init()：看三个 arena 如何按 memory plan 分配。
  2. bind_model()：看 weights arena 里每个参数 tensor 的 shape。
  3. init_kv_cache()：看 KV cache 的真实 shape。
  4. prefill() 和 decode_one()：看 workspace reset 和 cache 长度如何推进。
  5. run_layers()：看 token 到 logits 的完整主路径。
  6. CpuBackend::attention_out()：看当前 K/V 如何写入 cache，以及 attention 如何读取历史。
核心心智模型： 权重负责“怎么变换”，workspace 保存“这次正在算什么”，KV cache 保存“过去已经算过什么”。推理就是把这三类数据按层组织起来，让一个 token 一步步变成下一个 token 的概率分布。
